**版本**: P0-v1.0

## 0. 前置说明

本文档定义了从「快照式状态模拟器」升级为「可交互写实生活世界引擎」的三项 P0 架构改造。现有底座（事件总线、时序主循环、SQLite 持久化、Hook 探针、每日报告、编译验证体系）已足够扎实，三项改造全部以插件式接入，零破坏性。

补完 P0 后形成的标准扩展范式：

> 新增维度 = 挂载属性到自我实体 + 定义对应行为 + 配置预期落差规则 + 接入 Hook 与日报

## P0-1：自我实体模型（Self Entity）

### 1.1 核心定位

把分散在生理、亲密、空间模块里的「人」的属性，统一收敛为**世界中唯一的第一人称实体**。它是所有感知的接收者、所有行为的发出者、所有状态变化的承载者。

### 1.2 核心数据结构

| 分类 | 核心属性 | 对接现有模块 |
|------|----------|--------------|
| 基础身份 | 身高、体重、年龄、基础体质参数 | 复用 `physio_service` 基线 |
| 身体状态 | 当前姿势（站/坐/躺/行走/弯腰）、当前动作（打字/喝水/进食/发呆/睡眠）、视线方向、穿着状态、肢体疲劳分布 | 生理层 + 亲密引擎统一入口 |
| 空间状态 | 当前场景、场景内坐标、朝向、移动速度、与周边物件的距离 | 对接 `scene_service` / `collision_service` |
| 注意力焦点 | 当前关注的物件/人物/信息内容、注意力集中度、分心阈值 | 为后续信息感知、社交感知预留入口 |
| 内部状态 | 精力、疲劳、饥饿、情绪基底、健康度 | 统一所有生理/心理状态的出口 |

### 1.3 数据库

独立 `self_state` 表，每秒 tick 持久化，全时序不可逆：

```sql
CREATE TABLE IF NOT EXISTS self_state (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  tick INTEGER NOT NULL,
  timestamp TEXT NOT NULL DEFAULT (datetime('now','localtime')),
  -- 身体状态
  posture TEXT NOT NULL DEFAULT 'stand',     -- stand/sit/lie/walk/bend
  action TEXT NOT NULL DEFAULT 'idle',       -- 当前动作
  gaze_direction TEXT DEFAULT 'forward',
  clothing_state TEXT DEFAULT 'casual',
  limb_fatigue_json TEXT DEFAULT '{}',
  -- 空间状态
  current_scene TEXT NOT NULL DEFAULT 'home',
  position_x REAL DEFAULT 0,
  position_y REAL DEFAULT 0,
  position_z REAL DEFAULT 0,
  facing REAL DEFAULT 0,                     -- 朝向角度
  velocity REAL DEFAULT 0,
  -- 注意力
  focus_target TEXT DEFAULT NULL,
  focus_intensity REAL DEFAULT 0.5,
  distraction_threshold REAL DEFAULT 0.3,
  -- 内部状态
  energy REAL DEFAULT 100,
  fatigue REAL DEFAULT 0,
  hunger REAL DEFAULT 0,
  mood_baseline REAL DEFAULT 50,
  health REAL DEFAULT 100,
  -- 元数据
  last_behavior TEXT DEFAULT NULL,
  state_tags TEXT DEFAULT NULL
);
```

### 1.4 接入方式

1. 新增 `src/self_entity/self_entity_service.ts`
2. 现有模块不再各自维护「人」的属性，统一读写自我实体
3. 原有逻辑保留，只做数据挂载，零破坏性
4. Hook 探针新增 `self` 分组
5. 每日报告新增「自我状态全景」板块

## P0-2：行为系统 + 延迟后果链

### 2.1 核心定位

补齐「感知→决策→执行→世界变化→新感知」的完整闭环。行为是主体作用于世界的唯一入口，内置**延迟后果队列**实现行为的非即时、连锁后果。

### 2.2 行为三层分类

#### 即时行为
开灯、喝水、坐下、拿起物件 → 执行即生效，后果即时呈现

#### 持续行为
工作、睡眠、走路、做饭 → 有持续时长，过程中持续产生状态变化，可中断

#### 连锁行为
出门上班 = 换衣→出门→通勤→到岗→开始工作 → 由多个子行为按顺序组成，有完整时序

### 2.3 行为标准字段

```typescript
interface BehaviorEvent {
  id: string;                    // 行为ID
  actor: string;                 // 发起者（self）
  trigger_time: number;          // 触发时间戳
  estimated_duration: number;    // 预计时长(秒)
  status: 'pending' | 'executing' | 'completed' | 'interrupted';
  category: 'immediate' | 'sustained' | 'chain';
  sub_behaviors?: BehaviorEvent[]; // 连锁行为子项
  immediate_effects: Effect[];   // 即时后果
  delayed_effects: DelayedEffect[]; // 延迟后果
}
```

### 2.4 延迟后果链核心机制

采用**时间轮盘队列**挂载在主循环 tick 中：

- 例1：熬夜行为 → 即时：疲劳+20 → 12h后：免疫力下降 → 72h后：感冒概率提升 → 1周后：皮肤下降
- 例2：连续三天不出门 → 第2天：空气质量下降 → 第3天：垃圾堆积 → 第4天：社交孤独累积
- 例3：节食行为 → 即时：饥饿感上升 → 3天：基础代谢下降 → 1周：精力上限降低 + 易怒阈值下降

所有延迟后果写入 `delayed_effects` 表，到点自动触发，支持叠加和抵消。

### 2.5 接入方式

1. 新增 `src/behavior/behavior_service.ts` + `src/behavior/delayed_effects.ts`
2. 扩展 `action_handle.ts` 为标准行为入口
3. 内部走事件总线广播 `action.start / action.tick / action.finish`
4. 各模块订阅并处理对应领域的状态变化

## P0-3：预期-现实落差引擎

### 3.1 核心定位

不是独立业务模块，而是**挂载在所有感知维度上的通用计算中间件**。解决「只有现实值、没有预期值，情绪只有静态水平、没有波动来源」的问题。

### 3.2 双轨数据结构

每个可感知维度维护三组值：

| 值类型 | 来源 | 说明 |
|--------|------|------|
| 预期值 | 历史拟合（7/30天数据）、外部承诺（天气预报/约定/公告）、自我设定（计划/目标/期待） | 自动+手动双来源 |
| 现实值 | 各模块上报的真实状态 | 实时更新 |
| 落差值 | 现实值 - 预期值 | 正向=惊喜/超预期，负向=失望/挫败 |

### 3.3 核心计算规则

#### 强度分级
落差幅度 × 维度权重 = 情绪影响强度（无感/轻微/明显/强烈/冲击五级）

#### 累积效应
- 同一维度连续负向落差，强度逐次放大
- 连续正向落差产生适应衰减（边际愉悦递减）

#### 时序衰减
- 重大落差回落慢，日常落差回落快
- 衰减曲线：指数衰减，半衰期按事件等级配置

#### 交叉放大
- 多个维度同时出现负向落差 → 触发**复合状态迁移**
- 工作落空 + 约会爽约 + 下雨 → 从常规态进入「低落倦怠态」
- 整体情绪基线整体下移，而非三个独立负面值简单相加

### 3.4 接入方式

1. 新增 `src/common/gap_engine.ts`，放在 common 目录
2. 所有感知模块可调用，传入预期与现实值，返回标准化情绪影响因子
3. 输出统一接入 `mood_envi_service`，成为情绪系统核心输入源
4. 每日报告新增「落差事件统计」板块
