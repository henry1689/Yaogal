## 三、设计规范（逐模块）

### Phase 1 — 空间引擎 + 自然规则

#### 1.1 空间引擎 (perception_space/)

**设计规范:**
- 三固定场景：home(居家)、office(办公室)、outdoor(近郊户外)
- 每个场景包含：name、size_㎡、lighting(自然光/灯光/暗)、indoor/outdoor标识
- 场景切换必须带耗时：home↔office 30min，home↔outdoor 10min，office↔outdoor 15min
- 场景切换期间锁定，不可并发切换

**物件管理:**
- 每个物件：id、name、position{x,y,z}、state(完好/损坏/使用中/空/满)、last_modified
- 物件表：furniture/appliance/small_item/food_drink/consumable
- 物件状态变更触发 OBJECT_STATE_CHANGED 事件

**碰撞检测:**
- 玩家位置{x,y}仅在当前场景内计算
- 物件占据区域：原点+半径的圆形碰撞区
- 防穿墙：边界检测（场景size_㎡换算边长）
- 防悬空：z轴在允许范围（0=地面，>0需支撑物）

**验收标准:**
1. 场景列表返回三场景信息
2. 场景切换有耗时和状态锁定
3. 物件增删改查通过API
4. 碰撞检测返回bool和阻挡物名称
5. 移动指令返回新坐标或阻挡原因

**核验机制:**
- 单元测试：场景切换耗时精确到秒
- 单元测试：物件边界碰撞（站在物件上、移入物件内）
- 单元测试：场景边界碰撞（超出场景范围）
- 端到端：连续10次随机移动不穿墙不悬空

#### 1.2 物候规则 (natural_env/natural_rule/)

**设计规范:**
- 季节判定：春(3-5月)、夏(6-8月)、秋(9-11月)、冬(12-2月)
- 昼夜判定：日出6:00、日落18:00（简化，后续可配置经纬度）
- 季节联动：夏季降温需求、冬季取暖需求、春秋适宜
- 昼夜联动：夜间自动灯光模式
- 物候现象：花开(春)、落叶(秋)、积雪(冬)、蝉鸣(夏)

**验收标准:**
1. 根据月份正确返回季节
2. 根据小时正确返回昼夜状态
3. 季节昼夜联合影响环境描述
4. 物候现象与季节匹配

### Phase 2 — 极简理化 + 情绪环境联动

#### 2.1 极简物理 (simple_physics/basic_gravity/)

**设计规范:**
- 重力：松手→掉落(z=0)，推倒→滑落(沿推力方向)，跳跃→上升后回落到z=0
- 掉落时间：t = √(2h/g)，g=9.8，h为当前z高度
- 位移速度：步行1.4m/s、快走2.0m/s、跑步3.5m/s
- 碰撞响应：停止移动或弹回

**验收标准:**
1. 物体从高处掉落回到地面
2. 跳跃上升后回落到地面
3. 推倒物体沿方向滑落
4. 移动遇到阻挡停止

#### 2.2 极简化化 (simple_physics/simple_chem/)

**设计规范:**
- 水温冷却：牛顿冷却定律，T(t) = T_env + (T0 - T_env) * e^(-kt)，k=0.01/s
- 食物变质：三级（新鲜→微变→腐败），取决于温度（高温加速）和湿度
- 茶水浓度：初始100%，每次倒出减少浓度，注水稀释
- 燃烧消耗：可燃物总量逐秒减少，速率与通风相关
- 茶杯比：容量300ml，每次注水+200ml

**验收标准:**
1. 100°C水在25°C环境中冷却到50°C约需4分钟
2. 食物在30°C以上加速变质
3. 茶水浓度逐次降低
4. 燃烧速率与通风（窗口开关）相关

#### 2.3 情绪环境联动 (creature_law/mood_envi_link/)

**设计规范:**
- 光照→情绪：暗→困倦/压抑，亮→精神/开朗
- 温度→情绪：热→烦躁，冷→收缩，舒适→放松
- 天气→情绪：晴→轻快、阴→沉闷、雨→忧郁/安逸
- 声音→情绪：噪音→焦虑，安静→专注
- 气味→情绪：霉味→不适，花香→愉悦
- 输出：mood_modifier（-50到+50），叠加到生理情绪基底

**验收标准:**
1. 不同环境组合产生不同的mood_modifier
2. 极端值（持续噪音+高温+黑暗）产生显著负值
3. 适宜值（安静+温暖+明亮）产生正值

### Phase 3 — 时间农历 + 天气API

#### 3.1 农历节气算法

**设计规范:**
- 公历↔农历转换：基于农历数据嵌入
- 24节气：基于太阳黄经计算（每15°一个节气）
- 月相：新月→上弦→满月→下弦，约29.53天周期
- 四季：春分/夏至/秋分/冬至为分界
- 数据来源：嵌入1900-2100年农历数据JSON

**验收标准:**
1. 2026年春节日期准确（1月17日除夕）
2. 今日节气正确
3. 月相与天文一致
4. 公历转农历正确

#### 3.2 和风天气API对接

**设计规范:**
- API: QWeather v7，free订阅
- 缓存: 30分钟，夜间(00:00-07:00)不请求
- 限流: 每分钟最多1次请求
- 降级: 失败时用上次缓存，标记数据时效性
- 数据: 温度/体感温度/湿度/气压/天气描述/风力/能见度/AQI

**验收标准:**
1. 首次调用返回真实天气数据
2. 30分钟内重复调用返回缓存
3. 夜间调用不发送HTTP请求
4. 失败降级到缓存且标注

### Phase 4 — 生理完整时序 + 运维Hook

#### 4.1 伤病愈合完整时序

**设计规范:**
- 擦伤(scrape): 72h愈合，分6个阶段，每12h更新一次
- 轻伤(minor): 7天愈合，分7个阶段，每天更新
- 重伤(major): 30天愈合，分10个阶段，每3天更新
- 不可跳变：每个阶段线性过渡，不能瞬间恢复
- 天气影响：高温高湿加速感染风险（延长10-20%恢复时间）
- 休息加成：睡眠充足加速恢复10%

**验收标准:**
1. 擦伤72h后才能完全恢复
2. 轻伤7天按阶段渐进
3. 重伤30天不可缩短
4. 感染风险在极端环境中触发

#### 4.2 孕期完整时序

**设计规范:**
- 触发: 受孕事件→锁定孕周280天（40周）
- 早期(1-12周): 恶心/疲劳/情绪波动，体征不可见
- 中期(13-27周): 腹部隆起/胎动/精力恢复
- 晚期(28-40周): 行动不便/宫缩/准备分娩
- 硬性约束: 280天未满绝对不能分娩
- 周维度: 每7天递增一孕周
- 日维度: 体重/腹围/激素线性渐变

**验收标准:**
1. 触发后锁定280天
2. 每天属性按周线性渐变
3. 早中晚期体征不同
4. 预产期前不能分娩

#### 4.3 运维Hook探针

**设计规范:**
- 独立Hook分组: world_core/world_physio/world_perception/world_api
- 探针: 每30秒 ping 各模块健康状态
- 异常: 连续3次失败→报警，写入异常日志
- 恢复: 连续3次成功→解除报警
- 降级: 非核心模块异常不阻塞整体运行

**验收标准:**
1. 各模块健康检查正常通过
2. 模拟模块故障触发报警
3. 恢复后自动解除
4. 非核心降级不影响核心

### Phase 5 — 端到端集成 + 测试 + 日报

#### 5.1 端到端集成

**设计规范:**
- 启动流程：数据库→时间→天气→生理→感知→监控→API
- 事件总线：所有模块状态变更通过事件广播
- 世界循环：每秒tick驱动衰减、物理、化学、生理更新
- 感知快照：每10秒生成完整快照并持久化

**验收标准:**
1. 全模块启动10秒内完成
2. 60秒内无内存泄漏（GC平稳）
3. 感知快照正确生成
4. 事件总线消息不丢失

#### 5.2 日报生成

**设计规范:**
- 触发: 每日凌晨01:00自动生成
- 内容: 健康评分(0-100)、各模块状态、天气摘要、生理状态、异常统计、趋势对比、优化建议
- 格式: Markdown，存入 world_daily_reports 目录
- 保留: 最近30天日报

**验收标准:**
1. 凌晨1点自动生成
2. 报告格式完整
3. 存档30天
4. 异常在报告中标红

#### 5.3 单元测试

**覆盖目标:**
- 空间引擎: 场景切换/物件管理/碰撞检测
- 物理: 重力掉落/位移速度/碰撞响应
- 化学: 冷却曲线/变质阶段/茶水稀释
- 时间: 农历转换/节气计算/月相周期
- 生理: 伤病恢复阶段/孕期周推进
- 亲密: 触觉事件/唤起度曲线/高潮触发
- 总计: ≥50个测试用例

## 四、设计标准（全局）

1. **写实不可逆**: 所有时序规则不可回溯、不可快进、不可瞬移
2. **因果闭环**: 每个状态变化有前因后果
3. **持久化**: 所有状态变更必须落库
4. **事件驱动**: 模块间通信仅通过事件总线
5. **失败降级**: 非核心模块失败不影响核心时钟
6. **独立安全**: 不触碰太虚境代码、不外连公网（天气API除外）
7. **TypeScript严格**: noImplicitAny、strictNullChecks 全开
